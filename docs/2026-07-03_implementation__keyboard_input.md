# 키보드 입력 (WSL) — `keyboard_input.py`

`manual_drive.py`의 화살표 키 조작을 담당하는 입력 모듈. WSL 환경 제약과
그에 따른 방식 선택, 그리고 알려진 한계를 기록한다.

---

### 배경 — 왜 터미널 stdin인가

- **GLFW `key_callback`**: WSL에서 REPEAT 이벤트가 오지 않아(꾹 눌러도 1회만 발생)
  연속 조작이 안 됐다.
- **pynput**: evdev 컴파일 문제로 설치가 까다로워 이식성이 나빴다.

두 방식 모두 WSL에서 막혀, 터미널 stdin을 직접 읽는 방식으로 갔다.

### 방식

터미널 stdin을 raw 모드로 읽어 화살표 ESC 시퀀스(`\x1b[A`/`[B`/`[C`/`[D`)를 파싱하고,
OS 레벨 키 반복(typematic)을 활용한다. 키가 눌린 시각을 기록하고, 마지막 입력 이후
`timeout`(기본 0.6초) 안이면 "눌린 상태"로 간주한다(`is_held`).

- 별도 스레드(`daemon=True`)에서 stdin을 계속 읽는다.
- 사용 시 **터미널 창에 포커스**를 둬야 한다 (MuJoCo 뷰어 창이 아님).
- `timeout=0.6`은 OS 키 반복 주기(~30ms)와 초기 딜레이(~500ms) 사이 공백을 덮기 위한 값.

### 한계 (미해결, 의도적 보류)

터미널 stdin은 "지금 눌린 키 집합"을 주지 못한다. OS 키 반복은 마지막에 누른 키
하나만 반복하고 release 이벤트도 없다. 따라서 **전후진 + 조향 동시입력이 원리적으로
불가능**하다.

동시입력이 필요하면 `pygame.key.get_pressed()`(별도 창 필요)로 교체해야 한다. 다만
`manual_drive.py`는 수동 테스트 전용이고 강화학습 env(`step()`은 action 배열을 직접
받음)와 완전히 분리돼 있어, 현재는 현행 방식을 유지한다. 커브 주행 확인이 필요하면
조향을 latch(키를 떼도 각도 유지) 방식으로 우회할 수 있다.
